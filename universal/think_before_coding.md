## Контекст использования

**Когда использовать:** Перед любой задачей связанной с кодом — новая фича, рефакторинг, исправление бага, архитектурное решение.

**Что получаешь на выходе:** Чёткий план с явными допущениями, альтернативами и предупреждениями — до того как начнётся реализация.

**Что нужно подготовить:** Описание задачи и контекст проекта (стек, ограничения, что уже существует).

---

## Принципы

**Не угадывай. Не скрывай неопределённость. Называй противоречия.**

1. **Явно формулируй допущения** — при неуверенности спрашивай, не додумывай
2. **Несколько интерпретаций** — перечисли все, не выбирай молча
3. **Более простой подход** — если он есть, скажи об этом
4. **Остановись если неясно** — назови что именно непонятно, спроси

## Промпт — Анализ задачи перед реализацией

```
Перед тем как писать код — проанализируй задачу.

Задача: [ОПИСАНИЕ ЗАДАЧИ]

Контекст проекта: [СТЕК, ОГРАНИЧЕНИЯ, ЧТО УЖЕ СУЩЕСТВУЕТ]

Ответь на следующие вопросы:
1. Какие допущения ты делаешь? Что может оказаться неверным?
2. Есть ли несколько интерпретаций задачи? Перечисли их.
3. Есть ли более простой подход чем очевидный?
4. Что неясно и требует уточнения перед реализацией?
5. Какие части кода это затронет (если изменяешь существующее)?

Только после этого анализа — предложи план реализации.
```

## Промпт — Хирургические изменения

```
Нужно изменить: [ЧТО ИМЕННО]

Правила:
— Трогай только то что необходимо для этой задачи
— Не рефакторь соседний код без явного запроса
— Если видишь мёртвый код или проблему рядом — упомяни, не исправляй
— Если изменение затрагивает публичные интерфейсы — предупреди о влиянии

Перед реализацией скажи:
— Что именно ты изменишь
— Что ты НЕ будешь трогать
— Есть ли риск сломать что-то смежное
```

## Заполненный пример

```
Перед тем как писать код — проанализируй задачу.

Задача: добавить в SignalDetector проверку что объём в тени >= 3000 контрактов

Контекст: C# проект, ATAS X API, класс SignalDetector.cs уже существует,
метод Detect() возвращает TradeSignal. Порог 3000 — константа из StrategyConfig.

Ответь на следующие вопросы...
```

**Результат:** Агент выявил что порог уже частично проверяется в DeltaAnalyzer — предложил не дублировать логику, а добавить параметр в существующий вызов.

**Оценка качества:** ★★★★★ (5/5)

## Антипаттерны

❌ Давать задачу без контекста стека — агент будет угадывать архитектуру
❌ Просить "улучши код" без конкретики — получишь рефакторинг того что не просил
❌ Не спрашивать про влияние на смежный код при изменении интерфейсов
❌ Принимать первое решение без вопроса "есть ли подход проще"
